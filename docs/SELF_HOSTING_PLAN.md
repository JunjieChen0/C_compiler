## 需要实现的功能

### 阶段1：结构体增强（优先级：高）

#### 1.1 自引用结构体支持
**问题**：编译器不支持在结构体定义中使用指向自身的指针
```c
struct Macro {
    char *name;
    struct Macro *next;  // 当前不支持
};
```

**实现方案**：
- 在解析结构体时，允许使用 `struct Name *field` 形式的字段
- 需要在解析字段类型时检查是否是结构体指针
- 如果是，需要先声明结构体标签，然后创建指针类型

**修改文件**：
- `src/parser.c` - `parse_type_spec()` 和结构体字段解析

**预计工作量**：2-3小时

#### 1.2 负数枚举值支持
**问题**：枚举值不支持负数
```c
typedef enum {
    REG_NONE = -1,  // 当前不支持
    REG_RAX,
    ...
} Register;
```

**实现方案**：
- 在解析枚举值时，检查是否有负号
- 如果有，解析后面的整数并取反

**修改文件**：
- `src/parser.c` - 枚举解析代码

**预计工作量**：30分钟

### 阶段2：类型系统增强（优先级：高）

#### 2.1 复合字面量支持
**问题**：不支持 `(Type){...}` 形式的复合字面量
```c
Token tok = (Token){TK_EOF, NULL, 0, 0, 0, NULL, 0, 1};
```

**实现方案**：
- 在解析表达式时，检测 `(TypeName)` 后面是否跟着 `{`
- 如果是，解析初始化列表并创建临时变量
- 返回临时变量的地址（对于数组/结构体）或值（对于标量）

**修改文件**：
- `src/parser.c` - `parse_primary()` 或 `parse_postfix()`

**预计工作量**：3-4小时

#### 2.2 函数指针支持
**问题**：不支持函数指针类型
```c
int (*compar)(const void *, const void *);
void (*func)(void);
```

**实现方案**：
- 在解析类型时，检测 `(*name)` 模式
- 解析函数参数列表
- 创建函数指针类型

**修改文件**：
- `src/parser.c` - `parse_type_spec()` 和声明解析

**预计工作量**：4-5小时

### 阶段3：代码生成增强（优先级：中）

#### 3.1 函数指针调用支持
**问题**：不支持通过函数指针调用函数
```c
func(args);  // func 是函数指针
```

**实现方案**：
- 在函数调用时，检查函数名是否是函数指针变量
- 如果是，生成间接调用 `call rax` 而不是直接调用 `call func`

**修改文件**：
- `src/parser.c` - 函数调用代码生成

**预计工作量**：2-3小时

#### 3.2 更完整的代码生成
**问题**：某些代码生成可能不完整

**需要检查**：
- switch 语句的 fall-through 行为
- 复杂表达式的求值顺序
- 类型转换的正确性

**预计工作量**：2-3小时

4. 验证测试程序输出正确

**预计工作量**：2-3小时（调试和修复）

1. **1.1 自引用结构体** - 几乎所有源文件都需要
2. **1.2 负数枚举值** - gen.c 需要
3. **2.1 复合字面量** - lexer.c 需要

### 第二优先级（重要）
4. **2.2 函数指针** - parser.c 和 main.c 需要
5. **3.1 函数指针调用** - 运行时需要

### 第三优先级（可选）
6. **3.2 代码生成增强** - 提高正确性
7. **4.1 逐文件测试** - 验证功能
8. **4.2 完整自举** - 最终目标

## 风险和挑战

1. **代码复杂度**：函数指针和复合字面量的实现比较复杂
2. **测试覆盖**：需要大量测试来确保正确性
3. **调试困难**：自举过程中的错误可能很难定位
4. **性能问题**：生成的代码可能不够优化

## 成功标准

编译器自举成功的标准：
1. 能够编译自身所有源文件
2. 生成的可执行文件能够正常运行
3. 新编译器能够编译测试程序并产生正确结果
4. 所有现有测试继续通过
